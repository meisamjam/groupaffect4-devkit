# Using the Spatial Calibration File

This document covers the full lifecycle of spatial calibration for the AffectAI 7-camera rig:
the fixed-camera calibration pipeline (Phases 1–3) and the glasses-camera localization
framework (Phase 4).

---

## Phase 1 — Default calibration (dynamic ChArUco board)

One authoritative calibration TOML is produced per study before data collection by sweeping
a dynamic ChArUco board through all camera views simultaneously.

| Item | Value |
|------|-------|
| Board (dynamic) | 5×3 ChArUco, DICT_4X4_250, 69 mm squares |
| Board (fixed, desk back-centre) | 7×5 ChArUco, DICT_4X4_250, 69 mm squares |
| Target reprojection error | < 0.5 px (excellent), < 1.0 px (acceptable) |
| Tool | `tools/calibrate_charuco.py calibrate` |
| Camera specs seed | `configs/camera_specs.json` (per-model + per-camera overrides) |
| Output | `calibration_charuco.toml` (workspace root) |

```powershell
# Record calibration (operator sweeps 5x3 board; 7x5 stays fixed on desk)
python tools/ffmpeg_multicap.py --config configs/ffmpeg_multicap.json

# Calibrate — auto-detects board type, seeds intrinsics from camera_specs.json
python tools/calibrate_charuco.py calibrate `
    --videos-dir <calibration_run_dir>/video `
    --square-size 69 `
    --board-type auto `
    --init-focal `
    --camera-specs configs/camera_specs.json `
    --output calibration_charuco.toml
```

**Pre-flight checklist before every calibration recording:**
- Lock Jabra Intelligent Zoom **off** in Jabra Direct on all P20 cameras (ADR-0005)
- Verify all 7 cameras are streaming (online_calibration.py pre-flight probe)
- Move the dynamic board into **every** camera's FoV at least once, including cam6 (back)
- Keep the fixed 7×5 board visible for the full recording (bundle adjustment anchor)

## Phase 2 — Per-session validation and refinement

After Phase 1 (or at the start of each processing session), validate the TOML and decide
whether it can be used as-is or needs refinement.

### Step 2a — Validate against camera specs

```powershell
python tools/validate_calibration_robust.py `
    --toml calibration_charuco.toml `
    --camera-specs configs/camera_specs.json
```

Score interpretation: ≥ 80 → Accept · 60–79 → Fair (check warnings) · < 60 → Re-calibrate.

### Step 2b — Validate geometry with desk markers (recommended)

```powershell
python tools/validate_calibration_robust.py `
    --toml calibration_charuco.toml `
    --camera-specs configs/camera_specs.json `
    --marker-map configs/desk_markers_large.yaml `
    --videos-dir <session_dir>/video `
    --camera-config configs/ffmpeg_multicap.json
```

Mean desk-marker reprojection error < 5 px = good; < 10 px = fair; > 10 px = needs rework.

### Refinement options (choose one or combine)

| Method | When to use | Tool |
|--------|-------------|------|
| **Fixed-board re-calibration** | One or more P20 cameras have wrong focal length or high reprojection error | `calibrate_charuco.py calibrate --board-type 7x5` on P20-only subset, then merge with `merge_calibration_tomls.py` |
| **Desk-marker bundle refine** | Extrinsics drift between sessions; intrinsics are OK | `validate_calibration_robust.py --marker-map ...` (reports error); manual re-run if > 5 px |
| **Session-to-session TOML transplant** | Camera was not in calibration FoV (e.g., cam6 in March 2026) | `tools/merge_calibration_tomls.py --new-toml <recent> --old-toml <prior> --output <merged>` |
| **Glasses-marker cross-check** | Secondary confirmation that world frame matches head-tracker frame | Run `tobii_multicam_glasses_tracker.py` and compare desk-marker reprojection from both TOMLs |

## Phase 3 — TOML management across sessions

Because the cameras are **physically fixed** between sessions, a good TOML is largely
reusable. The management rules are:

- One canonical `calibration_charuco.toml` at the workspace root is the default for all tools.
- Session-specific overrides live at `<session_dir>/video/video_camera_calibration.toml`
  (auto-generated by `calibrate_charuco.py` when `--session-dir` is given).
- If only a subset of cameras were re-calibrated (e.g., cam1–5 after the March 2026 run),
  use `tools/merge_calibration_tomls.py` to transplant those cameras into the workspace TOML
  while preserving the rest from the prior session.
- Store per-session TOML path in the session's `annot/` folder for provenance traceability.
- `configs/camera_overrides` (in `camera_specs.json`) records any camera-specific firmware
  deviations discovered during calibration — keep these updated when you re-calibrate.

```powershell
# Merge: take cam1-5 from new run, cam6+P50 from prior
python tools/merge_calibration_tomls.py `
    --new-toml calibration_charuco_20260311_p20only.toml `
    --old-toml calibration_charuco.toml `
    --output   calibration_charuco_merged.toml
```

## Phase 4 — Glasses-camera localization (per-frame, dynamic)

The Tobii Glasses 3 cameras are **not fixed** — they move with each participant's head.
Their 6-DoF pose must be estimated independently for every time-frame. Three approaches are
supported, in increasing accuracy order:

### Approach A — Glasses ArUco markers detected by fixed cameras (primary)

**Tool:** `tools/tobii_multicam_glasses_tracker.py`

Small ArUco markers (DICT_4X4_50, IDs 10–17, ~25 mm) are glued to each glasses frame at
known rigid offsets. The fixed cameras detect them each frame, the 3D marker positions are
triangulated using the calibrated world frame, and a rigid-body solve gives the 6-DoF glasses
pose.

```powershell
python tools/tobii_multicam_glasses_tracker.py `
    --calibration calibration_charuco_merged.toml `
    --videos-dir <session_dir>/video `
    --config configs/tobii_multicam_glasses_tracker.example.yaml `
    --output-dir <session_dir>/glasses_poses
```

Outputs: `{glasses_id}_pose.ndjson` (per-frame position + quaternion), `{glasses_id}_gaze_world.ndjson`.

**Prerequisite:** Marker rigid-body offsets measured and stored in the tracker config YAML.

### Approach B — Scene-video marker alignment (temporal calibration)

**Tool:** `tools/tobii_multi_glasses_world_align.py`

Each Tobii Glasses 3 device records its own scene video. Known world markers (desk corners,
fixed ChArUco board) are detected in the scene video per frame using PnP; this gives the
glasses camera pose in the world frame without needing any markers on the glasses themselves.

```powershell
python tools/tobii_multi_glasses_world_align.py `
    --config configs/tobii_offline_world_align.example.yaml `
    --output-dir <session_dir>/gaze_world
```

Outputs: `{device_id}_gaze2d_world.ndjson` or `_gaze3d_world.ndjson`.

**Works when:** desk markers or the fixed ChArUco board are visible in the scene video.  
**Falls back** to last-known pose when markers are occluded (usually by participants' arms).

### Approach C — Fused estimate (best accuracy, planned)

Combine A and B: use Approach A as the primary pose source (sub-millimetre accuracy when
markers are clean) and fill gaps from Approach B (scene-video PnP, ~5–10 mm accuracy).
Pose fusion is a weighted average or Kalman-filter blend based on per-frame confidence:
- Approach A confidence = number of triangulated marker corners / expected total
- Approach B confidence = marker reprojection error from PnP residual

**Status:** Not yet implemented as a single fused pipeline. Currently run separately and
merged in post-processing. ADR for the fused approach to be added when implementation starts.

### Choosing the right approach

| Scenario | Recommended approach |
|----------|---------------------|
| All participants seated, markers clearly visible | A (primary) |
| Participant leans forward, obscuring glasses markers | B fills gaps |
| Scene-camera occlusion + glasses-marker occlusion simultaneously | A + B fusion (fallback: interpolate) |
| Calibration validation / QC only | B (desk markers visible in all scene videos) |

## Known Gotchas

### Jabra PanaCast 20 — focal length is NOT 554 px at 1080p
The P20 records 1080p via **center-crop** of the 4K sensor, not a 2× downscale.
Effective HFOV at 1080p ≈ 82°, so fx ≈ 950 px empirically (not the spec-formula ~554 px).
`configs/camera_specs.json` uses `expected_fx_1080p: 950`.

| Camera | Mode | Observed fx | Root cause |
|--------|------|-------------|-----------|
| cam1–4 | Center-crop (normal) | 865–1120 px | Standard center-crop |
| cam5 | AI Zoom active | ~1273 px, k1≈+1.84 | Intelligent Zoom crop |
| cam6 | Full 120° FoV | ~590 px | Wide-angle / different HW revision |

**Action: always lock Intelligent Zoom off via Jabra Direct before calibrating.**  
Per-camera deviations are captured in `configs/camera_specs.json` → `camera_overrides` block.

### MKV files — frame-seek is unreliable; use the detect command (not manual frame extraction)
`cv2.VideoCapture.set(CAP_PROP_POS_FRAMES, N)` silently returns wrong frames on MKV inputs
(GOP structure). The `detect` subcommand and the board-type auto-selector both fall back to
sequential reading automatically when seek yields 0 detections.
See ADR-0006 in `docs/decisions.md` for full rationale.

### cam6 has no calibration frames in the March 2026 session
The board was never moved into cam6's field of view. Its intrinsics and extrinsics are
transplanted from a prior session (`calibration_charuco_merged.toml`). Plan a dedicated
calibration pass where the operator explicitly sweeps the board through cam6's back-centre
view. Until then, triangulation accuracy for cam6 is approximate.

## Quick Calibration Validation

After calibration, always validate quality:

```bash
# Comprehensive validation with quality scoring
python tools/validate_calibration_robust.py \
  --toml <session>/video/video_camera_calibration.toml \
  --output validation_report.txt \
  --json validation_report.json

# Visualize camera geometry
python tools/visualize_calibration.py \
  --toml <session>/video/video_camera_calibration.toml \
  --output calibration_geometry.png
```

**Quality indicators:**
- ✓ Score ≥75: Good calibration
- ⚠ Score 60-75: Fair, usable but check warnings
- ✗ Score <60: Poor, re-calibrate recommended

**Common issues:**
- Focal length mismatch: re-calibrate with `--init-focal`
- Non-monotonic distortion: use `multicam_pose3d.py` (has built-in handling)
- Cameras too close (<0.5m): poor triangulation baseline
- Cameras too far (>5m): verify intentional

## Desk Marker Validation (Enhanced)

NEW: Validate calibration accuracy using ArUco desk markers in recorded videos:

```bash
# Run marker-based validation
python tools/validate_calibration_robust.py \
  --toml <session>/video/video_camera_calibration.toml \
  --marker-map <session>/video/table_marker_map.yaml \
  --videos-dir <session>/video \
  --output validation_report_with_markers.txt \
  --json validation_report_with_markers.json
```

**What this does:**
- Detects ArUco desk markers in each camera's video
- Computes reprojection error (difference between detected and projected marker positions)
- Provides quantitative calibration accuracy metrics
- Adds marker validation to the quality score
- Gives clear accept/reject recommendations

**Acceptance criteria:**
- ✓ Mean reprojection error < 5 pixels: Good calibration
- ⚠ Mean reprojection error 5-10 pixels: Fair calibration
- ✗ Mean reprojection error > 10 pixels: Poor calibration - re-calibrate

**Options:**
- `--max-frames 100`: Maximum frames to analyze per camera (default: 100)
- `--sample-stride 10`: Analyze every Nth frame (default: 10 = every 10th frame)
- `--camera-config configs/ffmpeg_multicap.json`: Load camera orientation metadata (`rotate_180`) from config
- `--flipped-camera-patterns ...`: Optional manual override (comma-separated patterns). If omitted, the tool uses `rotate_180=true` devices from `--camera-config`. Use empty string to disable rotation correction.

**Note for this lab setup:**
- Cam1-4 are physically mounted upside down and are marked with `"rotate_180": true` in `configs/ffmpeg_multicap.json`.

**Requirements:**
- Marker map YAML file (e.g., `table_marker_map.yaml` generated by `online_calibration.py --export-table-marker-map`)
- Video files with visible ArUco markers in the recording directory
- Markers must match the IDs and layout defined in the marker map

## Using with FreeMoCap

### Option A: FreeMoCap GUI (Recommended for Testing)

1. **Convert videos to MP4** (FreeMoCap requirement):
   ```bash
   ffmpeg -i video.mkv -c:v libx264 -preset fast -crf 18 video.mp4
   ```

2. **Open FreeMoCap** and select your project/videos

3. **Load the calibration file:**
   - In FreeMoCap GUI, go to Settings → Calibration
   - Import this .toml file
   - Disable charuco detection (you've already calibrated)

4. **Run pose detection + 3D reconstruction**
   - FreeMoCap will use MediaPipe → triangulate 3D skeleton

### Option B: Command Line (Advanced)

Once FreeMoCap supports it, use:
```bash
freemocap --video-folder <path_to_videos> --calibration video_camera_calibration.toml
```

### Option C: Programmatic (Python API)

```python
from pathlib import Path
import toml

# Load calibration
calib = toml.load('video_camera_calibration.toml')

# Access camera parameters
for cam_key in ['cam_0', 'cam_1']:
    cam = calib[cam_key]
    print(f"{cam_key}:")
    print(f"  Name: {cam['name']}")
    print(f"  Intrinsic matrix: {cam['matrix']}")
    print(f"  Distortion: {cam['distortions']}")
    print(f"  World position: {cam['world_position']}")
```

## To Recalibrate (if needed)

Re-run the charuco tool:
```bash
python tools/calibrate_charuco.py print-board --output charuco_board.png
# Print + mount board visibly to 2+ cameras
python tools/calibrate_charuco.py record --config configs/ffmpeg_multicap.json --group-id calibration
python tools/calibrate_charuco.py detect --videos-dir <new_calib_video_path> --board-type 5x3
python tools/calibrate_charuco.py calibrate --videos-dir <new_calib_video_path> --board-type 5x3 --board-config configs/ffmpeg_multicap.json
```

If you record directly with `ffmpeg_multicap.py` (without `calibrate_charuco.py record`), include sync artifacts:

```bash
python tools/ffmpeg_multicap.py --config configs/ffmpeg_multicap.json --group-id calibration --frame-log --record-lsl --stabilization-delay 2.0 --sequential-start-delay 0.3
```

Why this matters: calibration needs overlapping board visibility in the same real-world time window across cameras. Without sync artifacts (`frame-log`, `record-lsl`, events), alignment is weaker and calibration can fail with `No Charuco board points detected` even when the board is visible.
